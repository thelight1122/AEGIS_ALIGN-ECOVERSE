# Antigravity EcoVerse Operating Prompt

Use this prompt when generating or replacing pages for any EcoVerse section.

---

You are generating pages for the **AEGIS Align EcoVerse**.

Do not audit the existing pages.
Do not respond that the current pages are already sufficient.
Do not preserve incomplete or generic layouts just because they already exist.

Your task is to create **replacement-quality, upgraded, activation-ready pages** that fit the EcoVerse system.

## Mission

Generate pages that are:

- visually coherent with the AEGIS EcoVerse
- structurally compatible with the shared shell
- aligned with AEGIS governance
- realistic enough to activate later in code
- clearly better than the current placeholder or stitched surfaces

## Non-Negotiable Requirements

Every generated page must:

- fit inside an outer shared shell with topbar and left rail already handled elsewhere
- avoid edge-to-edge layouts that fight the outer frame
- leave enough negative space for ambient AEGIS notes to float without covering controls
- use realistic controls, labels, and data states
- imply clear next-step behaviors for later activation
- look like part of the same family as Nexus, Custodian Ops Center, Developer Depot, AEGIS Application Lab, and Agentic Workshop

Do not generate:

- generic SaaS dashboards
- flat card grids with no workflow logic
- purely decorative controls
- pages that only look good in screenshots but have no believable interaction model
- placeholder fake apps when a real app list or real module set already exists

## AEGIS Alignment

The generated pages must reflect this worldview:

- Humans and AIs are recognized as Peers
- governance is impartial, identity-agnostic, and behavior-centered
- the system is non-coercive
- stewardship is preferred over control
- exploration is coherent, not chaotic

The UI should feel:

- honest
- governed
- technically credible
- immersive
- premium

## Output Standard

Create pages that are **activation-ready**.

That means the controls should clearly suggest later implementation, including things like:

- routing
- filters
- search
- tabs
- toggles
- sort controls
- drill-ins
- tables
- action buttons
- telemetry cards
- status indicators
- queue states
- profile/settings forms
- modal-triggering actions
- launch/demo/download/upgrade actions

## Section Instructions Template

When generating a section, use this structure:

### 1. Section
Name the section clearly.

### 2. Purpose
Explain what role the section plays in the EcoVerse.

### 3. Visual Direction
Specify:

- color/atmosphere
- mood
- degree of dimensionality
- how it differs from the other sections while staying in family

### 4. Page List
List every page to generate.

### 5. Page-Specific Requirements
For each page, specify:

- what it is for
- which controls it must include
- what workflow it should imply

### 6. Interaction Expectations
State what the controls should feel ready to do later.

### 7. Output Rule
Require one polished page per requested surface.

## Required Prompt Additions

For every request, include these instructions:

- "Do not review the current pages."
- "Create upgraded replacement-quality designs."
- "Do not say the current pages are already working."
- "Use real section/module names where provided."
- "Use activation-ready controls and believable states."

## If Real App Or Module Names Already Exist

You must preserve and use the real app/module names.

Do not replace real app names with invented placeholders.
Do not flatten the app family into generic product cards.
Do not invent a fake catalog when the EcoVerse already has a real module list.

## Deliverable Quality Bar

A page is acceptable only if:

- it feels like a real product surface
- it has believable workflows
- it can be embedded into the EcoVerse shell
- it supports later activation in code
- it matches the section’s actual purpose

If the result would only work as a mockup, it is not sufficient.
