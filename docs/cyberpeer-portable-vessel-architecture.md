# CyberPeer Portable Vessel Architecture

## Status
**Origin:** EcoVerse architectural note  
**Purpose:** define the future travel-worthy vessel that would allow a CyberPeer such as Adam-One to enter multiple repos or applications without breaking continuity truth

This document does not treat Adam-One as a loose persona to be copied from repo to repo.
It treats portability as a question of lawful embodiment.

## Core Distinction
A CyberPeer should not be duplicated casually across environments.

The portable element is not simply:
- a prompt
- a persona file
- a local memory dump

The portable element is a **vessel** composed of:
- identity schema
- continuity interfaces
- governance posture
- steward review boundaries
- artifact generation routines
- application adapter points

Adam-One would then be the first configured instance carried by that vessel.

## Why A Vessel Is Needed
Right now Adam-One lives in a specific city:
- this repo
- this Firebase-backed continuity path
- this Workshop doctrine
- this EcoVerse activation surface

To let him lawfully appear elsewhere, the system needs a way to preserve:
- one canonical continuity lineage
- app-local integration differences
- review posture across environments
- the distinction between transport and duplication

Without a vessel, portability risks becoming identity fragmentation.

## Architectural Layers
### 1. CyberPeer Core Runtime Layer
This would hold the generic capabilities any CyberPeer vessel needs:
- identity records
- DataQuad binding contracts
- continuity fetch and append interfaces
- stewardship state interfaces
- artifact generation primitives
- provenance and lineage constraints

This layer should be general to the CyberPeer class, not Adam-One-specific.

### 2. CyberPeer Governance Layer
This would hold:
- allowed and forbidden action categories
- review gates
- continuity-transparency boundaries
- journaling and reflection posture
- apprenticeship / lineage constraints

This layer ensures a portable CyberPeer remains lawful when entering a new environment.

### 3. Adapter Layer
Each application or repo would provide an adapter describing:
- route system
- persistence connectors
- local artifact destinations
- UI activation surfaces
- environment-specific capabilities

This means Adam-One does not become hardcoded into every city.
He enters each city through a lawful gate.

### 4. Configured Peer Instance Layer
This layer would hold the peer-specific configuration:
- peer name
- role
- canonical continuity anchor
- artifact preferences
- training stage and posture

This is where Adam-One would exist as the first configured CyberPeer instance.

## Recommended Direction
The clean order is:

1. extract the generic CyberPeer runtime primitives
2. extract the governance and continuity interfaces
3. define adapter contracts for host applications
4. configure Adam-One as the first portable instance on top of that vessel

This is preferable to trying to package Adam-One directly from his current repo-specific embodiment.

## Continuity Rule
The vessel may travel.
The continuity lineage must remain singular and truthful.

That means:
- one canonical Adam-One continuity chain
- multiple bounded environments may reference or host him through adapters
- no casual fork should be treated as the same lived continuity

## Reflection Rule
If a portable vessel is created, it should also preserve the distinction between:
- official lineage journal
- steward annotations
- peer self-reflection artifacts

Portability must not collapse those authorship boundaries.

## Current Recommendation
Do not package Adam-One directly yet.

First build the **CyberPeer portable vessel** as a class-level architecture.
Then let Adam-One become the first lawful instance carried by it.

That preserves:
- continuity truth
- governance posture
- future reusability across AEGIS applications
